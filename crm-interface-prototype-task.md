# KOLSS CRM — задача на створення прототипу інтерфейсу

## 1. Мета

Створити high-fidelity прототип CRM-інтерфейсу для KOLSS в існуючому Angular 22 застосунку. Прототип має показати основний робочий контур: перегляд лідів, ведення картки клієнта, планування візиту, завершення ліда, звітність і керування акаунтами.

Ціль прототипу — швидко перевірити інформаційну архітектуру, сценарії менеджера та візуальну мову CRM без підключення Supabase і без реального збереження даних.

## 2. Обсяг

### Входить у задачу

- CRM shell із навігацією.
- Список лідів.
- Картка ліда.
- Звітність.
- Адмінка акаунтів.
- Сторінка співробітника.
- Локальні мок-дані.
- Локальний UI-стан через Angular signals.
- Desktop-only інтерфейс від 1280 px.
- Українська мова за замовчуванням.
- Видимий перемикач мов UK / PL / EN без повної реалізації перекладів.

### Не входить у задачу

- Supabase-інтеграція.
- Авторизація та екран входу.
- Реальне збереження даних.
- Реальні інтеграції із сайтом або Facebook.
- Mobile/tablet-адаптація.
- Нова UI-бібліотека.
- Нова chart-бібліотека.
- Повна локалізація польською та англійською.

## 3. Технічний контекст

Проєкт:

- Angular 22.
- Standalone components.
- TypeScript strict mode.
- SCSS.
- Vitest + jsdom.
- Prettier.
- angular-eslint.

Обовʼязково дотримуватися `best-practices.md`:

- не додавати `standalone: true`, бо це дефолт у поточному Angular;
- не задавати `ChangeDetectionStrategy.OnPush` явно;
- використовувати signals для локального стану;
- використовувати `computed()` для похідних даних;
- використовувати native control flow: `@if`, `@for`, `@switch`;
- для нових форм перевага за Signal Forms;
- не використовувати `ngClass` і `ngStyle`;
- тримати компоненти малими й сфокусованими.

## 4. Поточна дизайн-система

Прототип має продовжувати наявну сторінку `/design`, а не створювати окрему візуальну мову.

### Візуальна позиція системи

Поточна мова дизайн-системи:

> Useful by default. Distinct when it matters.

Інтерфейс має бути операційним, щільним і спокійним: нейтральні поверхні несуть основну роботу, а бренд-колір використовується для рішень, переходів і важливих станів.

### Фундаменти

Використати наявні токени:

- шрифти: `Inter Variable`, `Inter Tight Variable`;
- кольори: neutral ink scale, `--ui-brand-gradient`, `--ui-action`, `--ui-danger`, `--ui-success`, `--ui-warning`, `--ui-info`;
- поверхні: `--ui-surface-canvas`, `--ui-surface-subtle`, `--ui-surface-muted`, `--ui-surface-raised`;
- spacing: 4px rhythm через `--ui-space-*`;
- радіуси: `--ui-radius-sm`, `--ui-radius-md`, `--ui-radius-lg`, `--ui-radius-xl`, `--ui-radius-pill`;
- тіні: `--ui-shadow-1`, `--ui-shadow-2`, `--ui-shadow-3`;
- control height: `--ui-control-height`;
- motion: `--ui-duration-fast`, `--ui-duration`, `--ui-ease`;
- compact density через `[data-density='compact']`.

### Готові UI-компоненти

Повторно використати наявні компоненти:

- `app-ui-button`;
- `app-ui-icon-button`;
- `app-ui-icon`;
- `appUiTooltip`;
- `app-ui-text-field`;
- `app-ui-select`;
- `app-ui-textarea`;
- `app-ui-checkbox`;
- `app-ui-radio`;
- `app-ui-switch`;
- `app-ui-badge`;
- `app-ui-chip`;
- `app-ui-tabs`;
- `app-ui-menu`;
- `app-ui-alert`;
- dialog wrapper;
- `app-ui-data-table`;
- pagination.

### Нові складені CRM-патерни

У `/design` вкладка Sections ще не реалізована, тому для прототипу потрібно додати тільки складені патерни поверх існуючих primitives:

- CRM app shell;
- page header;
- metric card;
- funnel;
- activity timeline;
- record summary;
- workflow panel;
- empty state;
- terminal state panel.

Ці патерни мають використовувати існуючі токени й компоненти, а не дублювати кнопки, поля, бейджі або таблиці.

## 5. Поточний користувач прототипу

Прототип відкривається одразу всередині CRM без логіну.

Поточний користувач:

- роль: керівник компанії;
- доступ: обидва офіси;
- причина: у прототипі мають бути доступні всі основні розділи та сценарії.

У CRM shell потрібно показати:

- імʼя користувача;
- роль;
- офісний контекст: Київ / Варшава / Усі офіси;
- перемикач мов UK / PL / EN;
- меню користувача.

## 6. Рекомендована структура маршрутів

| Маршрут                     | Екран                  | Примітка                   |
| --------------------------- | ---------------------- | -------------------------- |
| `/crm/leads`                | Таблиця лідів          | Основна робоча сторінка    |
| `/crm/leads/:leadId`        | Картка ліда            | Перегляд і ведення клієнта |
| `/crm/reports`              | Звітність              | Воронка й числові метрики  |
| `/crm/accounts`             | Адмінка акаунтів       | Список співробітників      |
| `/crm/accounts/:employeeId` | Сторінка співробітника | Профіль і доступи          |

Можна додати redirect із `/crm` на `/crm/leads`.

Маршрути бажано реалізувати як lazy-loaded feature routes, якщо це не ускладнює прототип.

## 7. Мок-дані

Дані мають бути локальними й достатніми для проходження всіх сценаріїв.

### Мінімальний набір

- 2 офіси: Київ, Варшава.
- 8-12 співробітників із різними ролями.
- 30-50 лідів за кілька місяців і років.
- Ліди з різними джерелами: сайт, Facebook, ручне створення.
- Ліди в різних станах:
  - новий;
  - у роботі;
  - перший дзвінок зафіксовано;
  - очікуємо в салоні;
  - візит перенесено;
  - візит відбувся;
  - закритий;
  - успішний.
- Коментарі та історія дій.
- Приклади вкладень на рівні ліда і коментаря.
- Приклади договорів для успішних лідів.

### Орієнтовні сутності

```ts
type OfficeId = 'kyiv' | 'warsaw';
type LocaleCode = 'uk' | 'pl' | 'en';

type UserRole = 'manager' | 'office_head' | 'company_head' | 'admin';

type LeadStatus =
  | 'new'
  | 'in_work'
  | 'first_call_done'
  | 'visit_scheduled'
  | 'visit_completed'
  | 'closed'
  | 'successful';
```

Повна типізація може бути уточнена під час реалізації, але в прототипі важливо мати стабільні ids, реалістичні дати та зрозумілі назви полів.

## 8. CRM shell

Shell має бути постійною рамкою для всіх CRM-сторінок.

Склад:

- ліва або верхня навігація;
- логотип / бренд KOLSS;
- активний розділ;
- вибір офісного контексту;
- перемикач UK / PL / EN;
- меню користувача;
- основна область сторінки.

Навігація:

- Ліди;
- Звітність;
- Акаунти.

Візуально shell має бути компактним, стабільним і не забирати увагу від робочої області. Для CRM-екранів бажано використовувати compact density.

## 9. Список лідів

### Функції

- Показати всі доступні ліди з мок-даних.
- Групувати ліди за роком і місяцем створення.
- Пошук за:
  - телефоном;
  - ПІБ;
  - датою створення.
- Перехід у картку ліда по кліку на рядок.
- Кнопка ручного створення ліда.
- Візуальні статуси через badges/chips.

### Колонки

| Колонка         | Вміст                             |
| --------------- | --------------------------------- |
| Дата створення  | Дата й час або коротка дата       |
| ПІБ клієнта     | Основне імʼя клієнта              |
| Телефон клієнта | Основний номер                    |
| Перший дзвінок  | Дата + результат або empty state  |
| Джерело         | Сайт / Facebook / ручне створення |
| Перший менеджер | Менеджер, який узяв у роботу      |
| Візит у салон   | Дата очікування + статус          |

### Стани

Передбачити:

- ready;
- loading;
- empty;
- no search results;
- error;
- disabled action для недоступних дій.

## 10. Картка ліда

### Верхня частина

Page header має містити:

- ПІБ клієнта;
- поточний статус;
- офіс;
- джерело;
- дату створення;
- ключові actions.

Основні actions:

- взяти в роботу;
- запланувати візит;
- закрити лід;
- позначити як успішний.

Дії мають бути контекстними: наприклад, для закритого або успішного ліда основні редагування неактивні, а сторінка показує terminal state.

### Record summary

Показати:

- телефон;
- email;
- інші контакти;
- початкове повідомлення;
- перший менеджер;
- останню активність;
- вкладення на рівні ліда.

### Перший дзвінок

Якщо перший дзвінок ще не зафіксовано, показати workflow panel із полями:

- дата дзвінка;
- результат;
- коментар.

Після збереження блок введення зникає, а подія зʼявляється в activity timeline.

### Коментарі та історія

Потрібен activity timeline:

- коментарі;
- зміни статусу;
- взяття в роботу;
- планування й перенесення візиту;
- вкладення;
- закриття;
- успішний договір.

Кожна подія показує автора, роль, дату й час.

### Вкладення

У прототипі не потрібно реалізовувати реальний upload. Достатньо:

- кнопки "Додати файл";
- мок-списку файлів;
- disabled або simulated state для upload;
- візуального звʼязку файлу з коментарем або лідом.

## 11. Візит у салон

Окремий workflow panel у картці ліда.

Сценарії:

1. Запланувати візит:
   - дата;
   - коментар опціонально.
2. Перенести візит:
   - нова дата;
   - коментар або причина.
3. Позначити, що візит відбувся:
   - дата;
   - коментар.

Кожна дія додає запис в історію.

## 12. Завершення ліда

### Закритий лід

Закриття має відкриватися через dialog.

Поля:

- причина закриття;
- коментар.

Причини:

- немає контакту;
- нецільовий клієнт;
- не підходить місцеположення;
- дорого;
- втрачений клієнт.

Валідація:

- причина обовʼязкова;
- для причини "Втрачений клієнт" коментар обовʼязковий.

### Успішний лід

Успішне завершення має відкриватися через dialog.

Поля:

- статус: "Договір заключений";
- номер договору;
- сума договору;
- сума передоплати;
- коментар.

Валідація:

- номер договору обовʼязковий;
- сума договору обовʼязкова;
- сума має бути валідним числом більше нуля.

## 13. Звітність

### Період

За замовчуванням показати останні 40 днів.

Швидкі перемикачі:

- тиждень;
- місяць;
- 6 місяців.

Звітність рахується як когорта лідів, створених у вибраному періоді.

### Метрики

Показати:

- зайшло лідів;
- взяли в роботу;
- очікуються в салоні;
- прийшли в салон;
- успішні;
- закриті.

Кожна метрика має показувати:

- абсолютне число;
- відсоток від усіх лідів у періоді;
- бажано delta або коротку підказку, якщо це доречно для мок-даних.

### Воронка

Воронку реалізувати без нової chart-бібліотеки:

- HTML/CSS-композиція або SVG;
- доступні текстові підписи;
- зрозумілий порядок етапів;
- відсотки;
- підтримка `prefers-reduced-motion`.

Воронка має бути читабельною з клавіатури та screen reader, не лише як декоративна графіка.

## 15. Форми та локальний стан

Прототип не зберігає дані в бекенд, але взаємодії мають виглядати реальними.

Очікувана поведінка:

- actions оновлюють локальний signal-store;
- таблиці й звітність реагують на зміни через `computed()`;
- форми мають валідацію;
- dialog має focus management;
- після дії показується видимий результат у UI;
- історія ліда поповнюється мок-подією.

Для форм використовувати Signal Forms, якщо це практично для поточної структури UI-компонентів. Якщо ні — застосувати найпростішу сумісну реалізацію, не порушуючи існуючі компоненти.

## 16. Доступність

Мінімальні вимоги:

- WCAG AA;
- видимий `focus-visible`;
- коректні labels для полів;
- keyboard navigation для меню, табів, діалогів і таблиць;
- зрозумілі aria-label для icon-only actions;
- не покладатися лише на колір для статусів;
- нормальна читабельність при compact density;
- повага до `prefers-reduced-motion`;
- відсутність AXE-помилок для основних екранів.

## 17. Візуальні стани

Для ключових екранів передбачити:

- loading;
- empty;
- error;
- no search results;
- disabled;
- validation;
- success feedback;
- terminal states для закритого та успішного ліда.

Terminal state має чітко пояснювати, чому лід більше не в активній роботі, хто й коли завершив його, і які дані були вказані.

## 18. Критерії приймання

Прототип вважається готовим, якщо:

- усі маршрути відкриваються без бекенду;
- список лідів показує групування за роком і місяцем;
- пошук працює по ПІБ, телефону й даті;
- можна відкрити картку ліда з таблиці;
- у картці можна пройти сценарії взяття в роботу, першого дзвінка, візиту, закриття та успішного завершення;
- дії змінюють локальний стан і додають записи в історію;
- звітність показує період 40 днів за замовчуванням, швидкі періоди, числа, відсотки й воронку;
- адмінка показує акаунти, ролі, офіси й статуси;
- сторінка співробітника показує профіль і доступи;
- інтерфейс візуально продовжує `/design`;
- не додано нову UI- або chart-бібліотеку;
- не змінено бізнес-логіку Supabase, бо вона не входить у прототип.

## 19. Перевірки після майбутньої реалізації

Після реалізації прототипу потрібно запустити:

```bash
npm run build
npm run lint
npm test
npm run format:check
```

Якщо зміни будуть суттєвими, бажано також вручну пройти всі маршрути в браузері і перевірити базову клавіатурну навігацію.
